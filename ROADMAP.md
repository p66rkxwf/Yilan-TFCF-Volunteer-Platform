# 開發路線圖（Roadmap）

## 1. 目前進度

### 志工端（已完成）
- 帳號註冊、登入、忘記密碼
- 瀏覽與搜尋志工活動（client-side Supabase 查詢）
- 活動詳情、收藏活動、送出報名
- 查看與取消個人報名紀錄
- 編輯個人資料、停用帳號
- FAQ / 服務條款 / 隱私政策 / 支援頁

### 管理端（已完成）
- 後台儀表板與報名概況
- 建立、編輯、取消、恢復活動
- 審核報名狀態（pending / approved / rejected / cancelled）
- 搜尋、篩選、管理使用者
- 查看個別志工資料與近期報名紀錄
- 最後登入時間追蹤（`profiles.last_login_at`）

### 資料模型現況
| 資料表 | 狀態 | 備註 |
|--------|------|------|
| `profiles` | ✅ 使用中 | 角色、狀態、社工指派、最後登入 |
| `activities` | ✅ 使用中 | 活動主檔，含 `cancel_deadline`、`capacity` |
| `registrations` | ✅ 使用中 | 報名狀態，**無簽到 / 時數欄位** |
| `favorites` | ✅ 使用中 | 唯一有完整 migration + RLS 的表 |

### 角色 / 列舉
- 角色：`system_admin`、`unit_admin`、`internal_staff`、`volunteer`、`guest`
- 報名狀態：`pending`、`approved`、`rejected`、`cancelled`
- 職務：`social_worker`、`general_staff`

### 已知缺口（現況限制）
- ⚠️ 只有 `favorites` 有提交完整 SQL migration；`profiles` / `activities` / `registrations` 的建表 SQL 未進 repo，無法從零建置
- ⚠️ `npm run lint` 在 Next.js 16 設定下會失敗
- ⚠️ 獎學金（`/scholarship`）僅有頁面入口，功能未開放
- ⚠️ 無自動化測試
- ⚠️ 無通知 / 提醒機制
- ⚠️ 無服務時數 / 簽到紀錄

## 2. 待新增功能（優先方向）

> 以下三項為已確認的優先推進方向，依建議實作順序排列。

### P1. 志工時數 / 簽到系統
**目標**：活動當天能記錄出席、累計服務時數，並產出服務證明。

需要：
- 資料模型：在 `registrations` 增加 `checked_in_at`、`hours`、`attendance`（present / absent / no_show）等欄位，或另開 `attendances` 表
- 管理端：活動當天的簽到名單（勾選出席、批次標記、補登）
- 志工端：個人中心顯示累計時數、各活動時數明細
- 服務證明：依時數產出可下載 / 列印的證明（PDF 或列印樣式頁）
- 與報名狀態連動：僅 `approved` 的報名可簽到

驗收：管理者能在活動結束後標記出席並填時數，志工能在個人中心看到累計時數並下載證明。

### P2. 通知 / 提醒系統
**目標**：報名審核結果、活動提醒等關鍵事件能主動觸達志工。

需要：
- 資料模型：新增 `notifications` 表（`user_id`、`type`、`title`、`body`、`read_at`、`link`）
- 觸發點：報名核准 / 退回、活動異動或取消、活動前 N 天提醒
- 站內通知：個人中心鈴鐺 + 未讀數，標記已讀
- Email 通知（第二階段）：透過 Supabase Edge Function 或外部寄信服務（Resend 等）
- 偏好設定：讓志工選擇要接收哪些通知

驗收：報名被核准 / 退回時，志工能在站內收到通知（並可選擇收到 Email）。

> 建議先做「站內通知」打底，Email 屬於需要外部服務與環境變數的第二階段。

### P3. 後台強化
**目標**：讓中心人員更有效率地經營與檢視平台。

需要（依價值排序）：
- 報表 / 匯出：報名名單、簽到名單、時數統計匯出成 CSV / Excel
- 儀表板數據：活動參與率、各區志工分布、時數總覽、近期趨勢
- 權限細分：落實 `system_admin` / `unit_admin` / `internal_staff` 的差異化權限（目前列舉已存在，需確認實際 enforcement）
- 操作稽核：關鍵管理操作的紀錄（誰在何時改了什麼）

驗收：管理者能匯出報名 / 簽到名單，並在儀表板看到關鍵營運指標。

## 3. 建議新增功能

> 非當前優先，但對平台完整度與營運價值高，建議排入後續迭代。

### 體驗 / 志工端
- **活動行事曆檢視**：以月曆呈現活動，補足現有列表式瀏覽
- **報名候補（waitlist）**：活動額滿後可候補，有人取消自動遞補
- **活動回饋 / 評分**：活動後讓志工填回饋，作為改善依據
- **志工徽章 / 里程碑**：依時數或參與次數給予成就，提升黏著度

### 營運 / 管理端
- **獎學金模組**（首頁已預留入口）：申請表單、文件上傳、審核流程、結果通知
- **批次溝通**：對某活動報名者群發站內通知 / Email
- **活動範本**：常態活動可從範本快速建立
- **志工分群 / 標籤**：依專長、地區、可服務時段分群媒合

### 平台 / 信任
- **稽核日誌**（同後台強化，可獨立成模組）
- **資料匯出 / 帳號刪除**（個資法 / 隱私合規）
- **無障礙與 RWD 檢視**：確認手機端報名與簽到流程順暢

## 4. 技術債與基礎建設

> 這些不是新功能，但會直接影響上面功能的開發速度與穩定度，建議優先補。

- [ ] **補齊核心 migration**：把 `profiles` / `activities` / `registrations` 與 enum 的建表 SQL 進 repo，達成「從零建置」
- [ ] **修復 lint**：改用 Next.js 16 相容的 ESLint 執行方式
- [ ] **自動化測試**：至少替報名 / 審核 / 簽到等關鍵流程加 e2e 或 server action 測試
- [ ] **統一資料存取**：確認志工端 client-side 查詢與管理端 server-side 的權限邊界（RLS 覆蓋率）
- [ ] **Email / 排程基礎**：通知系統與活動提醒會需要 Edge Function + 排程（pg_cron 或外部 cron），建議與 P2 一起評估

---

## 建議實作順序（總結）

1. **基礎建設先行**：補核心 migration、修 lint（解除「無法從零建置」的風險）
2. **P1 時數 / 簽到**：對家扶營運價值最直接，資料模型改動相對單純
3. **P2 通知（站內優先）**：先站內、再 Email
4. **P3 後台強化**：報表匯出 + 儀表板數據
5. 之後再評估建議清單中的獎學金模組、候補、行事曆等
